What is a mobile framework?

Mobile

  • Adaptable to screen-size
  • Lighter-weight, faster
  • Touch-friendly

Framework

  • Can be built upon
  • Includes decisions on style

Speaker Notes

Mobile frameworks are a way to create mobile-friendly site. Mobile-friendly can mean a number of things. Start with the screen resolution.

All browsers are designed to handle varying page widths. I can resize my browser and text will reflow. Mobile frameworks are designed with particular sizes and ratios in mind. They tend to not waste a lot of space when viewed on tiny screens.

Mobile friendly means generally lighter-weight. This implies less bandwidth, the files are smaller, and the markup is not as complicated. This also means that sites are faster to download which is important when you have poor connections as many mobile devices do.

Mobile friendly means touch-friendly. Buttons and form elements are large and easy to use with a finger. Usually there is some mechanism to tap into swipe, tap, and hold events.

A framework implies that you get something to build on. This something should take less work than building it from scratch. Part of that is you accept the skeleton provided.

Additionally, you generally accept some part of the styling, although all frameworks can be bent to your will. Some provide this functionaliy very simply, like the jQueryMobile ThemeRoller.

For an app

  • Baker Ski Report
  • Sample app with Phonegap

Speaker Notes

There are three primary uses for a mobile framework.

First, when designing a limited version of a mobile site. Two examples that come to mind are Amazon and eBay, although they were designed without jQueryMobile. Basically, when you want to give mobile users something that's easy to use but doesn't do everything the "real" site does.

Another use of mobile frameworks is for mobile-first sites. Something that was designed from scratch for mobile usage.

Finally, some mobile app developers are using HTML and JavaScript to make more cross- platform apps without having to substantially rewrite the code.

Speaker Notes

I don't like ending on a negative note, so I am going to reverse the order here

Cons:

  • It's not native - slower, less responsive
  • It's not native - have to fill in native capabilities with something else
  • Quirky - have to do add JavaScript calls that are not intuitive
  • Quirky - forced into using a particular style unless you really roll your own

Pros:

  • Easy to learn - Little learning curve, quick to iterate because of HTML5 nature
  • Avoid the app store - No approval, no machine debugging, any platform
  • Cross-platform/cross-device - write once, use everywhere
